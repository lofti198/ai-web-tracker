# AI Web Tracker — обзор UI/UX (текущее состояние)

Документ для передачи UX-эксперту на ревью/оптимизацию. Описывает
структуру интерфейса, модель данных и пользовательские сценарии
**как есть на данный момент** — не то, какими они должны стать.
Скриншоты вкладок вставляются автором отдельно (места отмечены ниже).

## Что это за приложение

Расширение для Chrome (Manifest V3): по клику "Scan now" обходит сайты
через реальные вкладки браузера (страница списка → ссылки на сущности →
страница каждой сущности), фильтрует найденное (ключевые слова,
минус-слова, AI) и копит результаты локально (IndexedDB), показывая их
в отдельной вкладке-панели самого расширения. Сканирование только
ручное — расписания/автозапуска в фоне нет.

Изначально задумывалось как обучающий воркшоп-шаблон ("возьми и допиши
парсер под свой сайт"), но по ходу разработки стало применяться и как
личный рабочий инструмент — часть решений в UI сделаны под это (см.
раздел "Открытые вопросы" в конце).

Панель — не попап и не боковая панель Chrome, а обычная вкладка браузера
(`chrome-extension://.../panel/panel.html`), открывается кликом по иконке
расширения в тулбаре. Повторный клик активирует уже открытую вкладку, а
не плодит новые.

## Структура: 5 вкладок внутри панели + 1 модальное окно

Навигация — горизонтальный ряд кнопок-вкладок вверху панели, без
фреймворков (чистый JS, показ/скрытие блоков).

```
[ Main ] [ Searches ] [ Filters ] [ History ] [ Setting ]
```

**Logs — не вкладка**, это модальное окно (`<dialog>`), которое
открывается кнопкой "Logs" на вкладке Main, рядом со "Scan now"/"Stop".
Раньше в этом документе Logs числился как 6-я вкладка — по факту в панели
только 5 вкладок, Logs живёт отдельно (описан ниже, после History).

---

### 1. Main

**[Скриншот: вкладка Main]**

Стартовый экран. Задача — управление процессом сканирования и обзор
самых свежих находок.

Сверху вниз:
1. **"Scan now" / Stop** — две кнопки.
   - **"Scan now"** — единственный способ запустить сканирование
     (никакого расписания/автозапуска в фоне нет). По клику: запрашивает
     недостающие разрешения на домены источников, затем делает ровно один
     прогон. Пока прогон идёт, кнопка неактивна и показывает "Scanning…".
   - **Stop** — отдельная кнопка, неактивна (disabled), пока сканирование
     не идёт. Активируется на время прогона; прерывает его на середине
     (движок обхода перестаёт открывать новые вкладки сущностей и
     возвращает уже собранное к этому моменту).
2. **Live-индикатор прогресса** (появляется только во время прогона,
   исчезает через ~4 сек после завершения) — спиннер + текст текущего
   шага ("Opening list page: ...", "Fetching entity 3/10: ...") и список
   последних ~6 событий под ним.
3. **Статистика**: только "Last run: <дата/время>". ("Total found: N"
   здесь нет — этот счётчик живёт на вкладке History, не на Main.)
4. **"Just found"** — список сущностей текущего/последнего прогона (не
   вся история). Использует тот же компонент карточки, что и вкладка
   History (см. ниже). Заголовок и список **полностью скрыты**, пока
   находок нет — раньше секция была видна всегда с текстом-заглушкой
   ("Nothing found in this scan yet."), это было заменено на "скрыто, пока
   нечего показать" по явному запросу. Секция скрывается в момент старта
   нового прогона и появляется, как только приходит первая карточка;
   заполняется ПО ХОДУ: карточки каждого источника появляются сразу, как
   только этот источник отфильтрован и сохранён — не нужно ждать, пока
   отработают все источники. Переживает закрытие/открытие панели (список
   id последнего прогона хранится отдельно, финальная сверка со списком
   происходит по завершении прогона — если после сверки карточек 0,
   секция тоже скрывается).

Вкладка НЕ содержит: список сохранённых поисков (вынесен на Searches),
фильтры (на Filters). Настройки интервала/расписания в приложении нет
вообще.

### 2. Searches

**[Скриншот: вкладка Searches]**

Список **сохранённых поисков** (`searches.saved` в `chrome.storage.local`)
+ управление ими. Это не список парсеров/источников из кода — на свежей
установке список пуст, пока пользователь не добавит хотя бы один поиск
сам (демо-поиски, которые раньше сюда предзаполнялись, были явно удалены
миграцией в `parsers/searches.js`, см. `REMOVED_DEMO_SEARCH_NAMES`).

- Заголовок "Saved searches" + кнопка **"+ Add search"** справа от
  него.
- Список карточек: для каждого сохранённого поиска — имя, тумблер
  **On/Off** (временно выключить поиск из сканирования, не удаляя),
  кнопка **"Edit"** (открывает тот же попап, предзаполненный текущими
  Name/List URLs — редактирование "на месте" есть, удалять и добавлять
  заново не нужно) и кнопка **"✕"** (удаление с confirm-диалогом). Все
  элементы этого списка ведут себя одинаково — нет отдельной категории
  "встроенных, неудаляемых" карточек. Под именем — список List URLs,
  каждая ссылка кликабельна, открывается в новой вкладке браузера.
- Клик "+ Add search" (или "Edit" на существующем поиске) открывает
  модальное окно (нативный `<dialog id="add-search-dialog">`) **только с
  двумя полями**:
  - **Name** — короткий текстовый идентификатор.
  - **List URLs** (textarea, одна ссылка на строку) — все URL должны
    принадлежать одному из уже поддерживаемых сайтов (домен сверяется по
    `URL.origin` с `parsers/searches.js`'s `SUPPORTED_SOURCES`).
  - Кнопки "Add search"/"Save changes" (текст меняется в режиме Edit) и
    "Cancel".
  - Никакого поля для вставки кода (`extractLinks`/`extractDetail`) и
    никакого шага "Allow User Scripts" **не существует** — это описание
    относилось к более раннему, никогда не выпущенному варианту дизайна
    (см. `docs/HANDOFF.md`, раздел "Two registries for one source-adding
    mechanism"). Если введённый URL на неподдерживаемом домене, попап
    показывает инлайн-ошибку `#add-search-error` со списком поддерживаемых
    доменов и ничего не сохраняет.
  - Новый домен становится поддерживаемым только правкой файлов в
    `parsers/` (см. README "Как добавить свой источник", постоянный
    путь) — через сам попап добавить произвольный сайт нельзя.

### 3. Filters

**[Скриншот: вкладка Filters]**

Три независимых блока, каждый со своим тумблером вкл/выкл:

1. **Keyword filter** — тумблер + textarea со словами через
   запятую/с новой строки. Подсказка: "элемент проходит, если заголовок
   содержит хотя бы одно из слов".
2. **Exclude words** — тумблер + textarea, аналогично, но по логике
   отбрасывания ("отбрасывается, если содержит любое из этих слов").
3. **AI filter** — тумблер + textarea "критерий отбора своими словами" +
   подсказка, что в модель передаётся ещё и сниппет описания и текст
   резюме с вкладки Setting (если заполнен — см. ниже, сейчас поле резюме
   скрыто) + инлайн-предупреждение, если тумблер включён, а API-ключ
   (на отдельной вкладке Setting) не заполнен.

Внизу — кнопка "Save" (одна на все три блока) и надпись "Saved",
появляющаяся на ~2 сек после сохранения.

### 4. Setting

**[Скриншот: вкладка Setting]**

(В более ранней версии этого документа вкладка называлась "Profile" — в
коде `data-tab-target="profile"` остался как внутренний id, но видимая
подпись вкладки — "Setting".)

Сейчас на вкладке реально видны и работают два блока:

1. **OpenAI API key** — password-поле + кнопка Show/Hide, настройка
   уровня аккаунта (используется только AI-фильтром на вкладке Filters).
2. **Scan limits** — тумблер "Limit maximum number of found jobs from one
   search" + число (по умолчанию 10): движок обхода продолжает открывать
   ссылки, пока не наберёт столько подходящих находок, либо пока не
   кончится список.

Каждое поле сохраняется самостоятельно (debounce на ввод / сразу по
change), отдельной кнопки "Save" на вкладке нет.

**Resume / CV — блок есть в коде, но сейчас скрыт.** Разметка в
`panel/panel.html` обёрнута в HTML-комментарий, поэтому на вкладке она не
рендерится. Как он задуман (по коду в комментарии и в
`panel/tab-profile.js`, который на отсутствие этих полей реагирует
null-safe и не падает):

- `<input type="file" multiple>` для вложения одного или нескольких
  файлов сразу (любой тип — например разные резюме под разные типы
  вакансий) + textarea для текста резюме. Вложенные файлы показаны
  списком: каждый — кликабельная ссылка с именем и размером + своя кнопка
  "✕" на удаление. В AI-фильтр реально уходит только текст из textarea —
  вложения только для собственной справки пользователя. `.txt`-файлы при
  выборе дописываются в textarea автоматически (не заменяют, а
  добавляются); PDF/DOCX — нет (не парсится без стороннего кода,
  конфликтует с архитектурой "без сборки/сторонних зависимостей").
- `chrome.storage.local`-поля `profile.cvText`/`profile.cvFiles` и
  использование в AI-фильтре по-прежнему активны — просто сейчас нет UI,
  чтобы их заполнить, кроме как значение, сохранённое ДО того как блок
  скрыли. См. `docs/HANDOFF.md` для деталей.

### 5. History

**[Скриншот: вкладка History]**

Вся накопленная база находок за всё время (в отличие от "Just found" на
Main, которая показывает только последний прогон).

- **Тумблер "Use history"** (вкл. по умолчанию) сверху, с подсказкой:
  "When enabled, found jobs are saved here and already known links are
  skipped during scanning. When disabled, history is not saved and not
  used for duplicate checks." Выключение отражается на движке обхода
  сразу (`chrome.storage.local['history.enabled']`, читается один раз в
  начале каждого прогона): при выключенной истории новые находки не
  пишутся в IndexedDB (эта вкладка не растёт между прогонами) и старые
  ссылки не отсеиваются как "уже виденные" — при следующем прогоне те же
  вакансии снова будут показаны как новые. Не задокументировано в более
  ранней версии этого файла — фича добавлена позже.
- "Total found: N" сверху.
- Кнопка "Clear history" (с confirm-диалогом).
- Список карточек находок (компонент общий с "Just found" на Main —
  `panel/item-card.js`), сортировка — новые сверху. Каждая карточка:
  - заголовок-ссылка (открывается в новой вкладке);
  - строка meta: `meta` (например компания/локация) · имя источника ·
    дата обнаружения;
  - если есть `details` (доп. поля источника, например company/
    location/tags) — ряд мелких тегов-бейджей;
  - если есть `description` — обрезанный сниппет (до ~220 символов, с
    многоточием);
  - бейдж(и) "почему сущность здесь" — по какому фильтру она прошла:
    "Keyword filter: react, remote" (если сработал фильтр по словам) и
    отдельно "AI filter: <пояснение модели>" (если сработал AI-фильтр);
    может не быть ни одного бейджа, если оба фильтра были выключены.
- Пустое состояние: "Nothing found yet. Go to the Main tab and click
  "Scan now"."

### Logs (модальное окно, не вкладка)

**[Скриншот: окно Logs]**

Технический журнал прогонов — для отладки (в первую очередь своих
парсеров/источников). Открывается кнопкой "Logs" на вкладке Main (рядом
со "Scan now"/"Stop"), не находится в общем ряду вкладок наверху.

- Кнопка "Clear logs".
- Список записей: время, уровень (INFO/ERROR, цветовое различие),
  источник (имя парсера) и текст сообщения — например "Opening list
  page: ...", "Found 12 link(s) across 1 list page(s)", "5 new (of 12),
  visiting 5", "Fetching entity 2/5: ...", "Failed to load ...".
- Пустое состояние: "No log entries yet."

## Модель данных одной найденной сущности

То, что хранится в IndexedDB и лежит в основе каждой карточки
(History / Just found):

| Поле | Обязательное | Описание |
|---|---|---|
| `id` | да | нормализованный URL (без utm-меток) — ключ дедупликации |
| `url` | да | ссылка на сущность |
| `title` | да | заголовок |
| `meta` | нет | короткая строка для списка (например "Компания · Локация") |
| `description` | нет | полный текст (для двухфазных источников — например текст вакансии); в UI показывается обрезанным |
| `postedAt` | нет | дата публикации НА САЙТЕ-источнике (не дата, когда МЫ её нашли) |
| `details` | нет | произвольный набор ключ-значение от конкретного источника (company/location/tags и т.п.) |
| `source` | да | имя источника (ключ из реестра парсеров) |
| `foundAt` | да | когда расширение сохранило запись (timestamp) |
| `matchedKeywords` | нет | какие слова из keyword-фильтра совпали (пусто, если фильтр выключен/не участвовал) |
| `aiVerdict` | нет | пояснение AI-фильтра, почему сущность подошла (только если AI-фильтр был включён) |

Схема намеренно generic — не привязана к теме "вакансии", хотя все 5
поддерживаемых на сегодня источников (`parsers/searches.js`'s
`SUPPORTED_SOURCES`: Hacker News, Remote OK, We Work Remotely, EU Remote
Jobs, aijobs.com) — про вакансии.

## Пользовательские сценарии

### Сценарий 1 — первая установка
1. Скачать/склонировать репозиторий, включить Developer mode в
   `chrome://extensions`, "Load unpacked" → выбрать папку.
2. Кликнуть по иконке расширения → открывается новая вкладка-панель, по
   умолчанию на вкладке Main.
3. Main пустая: "Scan now" активна, Stop — нет, "Just found" полностью
   скрыта (не просто пустая — секции с заголовком нет вообще, пока
   нечего показать). "Total found: 0" — на вкладке History, не на Main.
4. Вкладка Searches тоже пустая: демо-поиски не предзаполняются на
   свежей установке (были явно вычищены миграцией в `parsers/searches.js`
   — см. `REMOVED_DEMO_SEARCH_NAMES`). Клик "Scan now" без единого
   сохранённого поиска ничего не находит — сначала нужен Сценарий 3.

### Сценарий 2 — запуск сканирования (после того как есть хотя бы один сохранённый поиск)
1. Пользователь на Main нажимает "Scan now".
2. Браузер один раз спрашивает разрешение на домены источников, на
   которые указывают сохранённые (и включённые, тумблер On) поиски — если
   ещё не выданы.
3. Кнопка "Scan now" становится неактивной и показывает "Scanning…",
   Stop становится активной.
4. Появляется live-индикатор: видно, как поочерёдно открываются вкладки
   списков, затем вкладки отдельных сущностей (мелькание вкладок в
   браузере — ожидаемо).
5. Источники обходятся параллельно; как только КАЖДЫЙ из них
   отфильтрован и сохранён — его карточки сразу появляются в "Just
   found" (секция становится видимой на первой же карточке), не дожидаясь
   остальных источников (если один источник медленнее другого, его
   результаты просто появятся позже).
6. По завершении всего прогона индикатор через ~4 сек скрывается, кнопка
   "Scan now" снова активна.
7. Дальше сканирование не повторяется само по себе — только по
   следующему клику "Scan now" (в т.ч. на другой день).

### Сценарий 3 — добавление источника из числа уже поддерживаемых (без правки файлов)
1. Переход на вкладку Searches → "+ Add search".
2. Заполнение формы: **только** Name и один или несколько List URLs (все
   на одном из уже поддерживаемых доменов — Hacker News, Remote OK, We
   Work Remotely, EU Remote Jobs, aijobs.com). Никакого кода вставлять не
   нужно — извлечение данных для этих доменов уже захардкожено в
   `parsers/searches.js`.
3. Сохранение → если домен(ы) поддерживаются, сразу запрашивается
   разрешение на новый домен (системный диалог Chrome); если нет —
   инлайн-ошибка в попапе со списком поддерживаемых доменов, ничего не
   сохраняется.
4. Новый поиск появляется в списке на вкладке Searches; при следующем
   клике "Scan now" участвует в сканировании (если тумблер On).

Добавление источника на **новом**, пока не поддерживаемом домене — это
отдельный, более тяжёлый сценарий: правка файлов в `parsers/` (см.
README "Как добавить свой источник"), а не что-то, что можно сделать
целиком из этого попапа.

### Сценарий 4 — настройка фильтров
1. Filters → включить нужный тумблер(ы), заполнить поля.
2. Save → надпись "Saved" на 2 сек.
3. Настройки применяются к СЛЕДУЮЩЕМУ прогону (не задним числом к уже
   сохранённым записям).

### Сценарий 5 — просмотр и разбор результатов
1. "Just found" на Main — что нашлось только что.
2. History — вся накопленная база, с теми же карточками, есть возможность
   очистить всё разом.
3. На карточке видно, по какому фильтру сущность прошла (бейдж) — полезно
   при настройке/отладке фильтров.

### Сценарий 6 — остановка на середине
1. Пока прогон идёт, пользователь жмёт Stop.
2. Движок обхода прекращает открывать новые вкладки сущностей; уже
   собранные к этому моменту сущности сохраняются как обычно.
3. Кнопка "Scan now" снова активна — можно запустить заново в любой
   момент (в т.ч. сразу же).

### Сценарий 7 — отладка своего источника
1. Источник добавлен, но в History/Just found пусто.
2. Пользователь идёт на Logs, видит пофазный журнал: сколько ссылок
   найдено на странице списка → сколько из них новых → успех/ошибка по
   каждой сущности.
3. Если проблема в самом List URL или имени поиска — правится через
   кнопку "Edit" на вкладке Searches (открывает тот же попап,
   предзаполненный; удалять и добавлять заново не нужно). Если проблема в
   селекторах (сайт не поддерживается или парсер собирает не то) —
   правка/добавление файла в `parsers/` (`extractLinks`/`extractDetail`)
   и перезагрузка расширения в `chrome://extensions`; попап "Add search"
   тут не поможет, у него нет доступа к коду парсера.

## Открытые вопросы / шероховатости (на мой взгляд, стоит обсудить с UX)

- **"Just found" vs "History"** — визуально почти идентичны (общий
  компонент карточки), различаются только временны́м охватом (последний
  прогон vs всё время). Не до конца очевидно из UI, что это разные вещи,
  пока не прочитаешь подписи.
- **5 вкладок + модалка Logs** — по мере роста функциональности число
  вкладок растёт (было 2, стало 5, плюс два модальных окна — Logs и
  add-search). Setting уже сейчас "готов" принять обратно скрытый блок
  Resume/CV (см. раздел 4) — если его вернуть, вопрос про склейку двух
  несвязанных вещей на одной вкладке снова станет актуальным.
- **Пустая вкладка Searches на свежей установке** — без демо-поисков
  (сознательно вычищены) пользователь должен сам знать, что добавить
  Search — обязательный первый шаг перед "Scan now". Ничего в UI Main это
  явно не подсказывает (кнопка "Scan now" активна и кликабельна даже без
  единого сохранённого поиска).
- **Список поддерживаемых доменов не виден заранее** — попап "+ Add
  search" сообщает, какие домены поддерживаются, только ПОСЛЕ того, как
  пользователь ввёл неподдерживаемый URL и получил ошибку. Нет списка/
  подсказки в самой форме до этого момента.
- **Мелькание вкладок при сканировании** — осознанный архитектурный
  выбор (реальные вкладки вместо фонового fetch), но с точки зрения UX
  это заметное, потенциально отвлекающее побочное явление, особенно если
  источников/сущностей много.
- **История можно выключить, но эффект не всегда очевиден** — тумблер
  "Use history" на вкладке History молча меняет поведение дедупликации
  при следующем прогоне (без истории те же вакансии находятся заново
  каждый раз); подсказка под тумблером это объясняет, но легко пропустить
  при беглом просмотре.
- **Нет расписания вообще** — сознательное упрощение (было — убрали по
  явной просьбе): каждое сканирование нужно запускать вручную, в т.ч.
  повторные проверки "не появилось ли новое" полностью на пользователе.
  Стоит обсудить, не станет ли это узким местом при реальном
  использовании (легко забыть перезапустить).
