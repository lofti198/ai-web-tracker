# AI Web Tracker

Расширение для Chrome (Manifest V3), которое по клику обходит сайты через
реальные вкладки браузера (страница списка → ссылки на сущности →
страница каждой сущности), фильтрует найденное (по ключевым словам,
минус-словам и опционально через AI) и показывает новые находки в панели.
Сканирование только ручное — по кнопке "Scan now", никакого расписания
или автозапуска в фоне нет.

Никакого билд-шага — ни Node.js, ни npm, ни сборщиков. Просто загружаете
папку в Chrome и работаете.

## Установка (5 шагов)

1. Скачайте или склонируйте этот репозиторий.
2. Откройте `chrome://extensions`.
3. Включите переключатель **"Режим разработчика"** (правый верхний угол).
4. Нажмите **"Загрузить распакованное расширение"** и выберите папку `ai-web-tracker`.
5. Кликните по иконке расширения — откроется вкладка панели. Готово.

## Как это работает

- Панель открывается **обычной вкладкой Chrome** (`chrome.tabs.create`), а
  не всплывающим окном и не встроенной боковой панелью. Повторный клик по
  иконке не плодит новые вкладки — активирует уже открытую.
- Сканирование идёт **через реальные вкладки браузера**, а не тихий
  `fetch()` из фона: расширение открывает страницу списка в фоновой
  вкладке, инжектит скрипт, который достаёт ссылки на сущности
  (`extractLinks`), и — если у источника есть детальные страницы — по
  очереди открывает каждую новую ссылку и достаёт с неё полные данные
  (`extractDetail`). Уже известные (ранее сохранённые) ссылки повторно не
  открываются. Это значит, что при сканировании в углу экрана будут
  ненадолго мелькать вкладки — это ожидаемое поведение, а не баг.
- **Main** — кнопка **"Scan now"** (запрашивает нужные разрешения и
  делает один прогон; пока прогон идёт, показывает "Scanning…" и
  недоступна для повторного клика) и отдельная кнопка **Stop** (активна
  только пока прогон идёт — прерывает его на середине, уже собранное к
  этому моменту сохраняется), live-индикатор текущего прогона и секция
  **"Just found"** — результаты текущего/последнего прогона (не вся
  история — она открывается кнопкой History), той же карточкой, что и в History.
  Каждый источник фильтруется и сохраняется сразу по завершении своего
  обхода, не дожидаясь остальных источников — карточки появляются в
  "Just found" по ходу сканирования, а не одним куском в самом конце.
- **Searches** — список сохранённых поисков (имя + реальные list-URL,
  каждый кликабелен) и кнопка **"+ Add search"** для добавления поиска
  прямо из панели (см. ниже).
- **Filters** — три независимых фильтра (по словам, по минус-словам, AI),
  каждый включается своим тумблером. Ключ OpenAI для AI-фильтра теперь
  вводится не здесь, а на вкладке "Setting".
- **Setting** — OpenAI API-ключ и резюме/CV (файл-вложение для справки + текстовое поле,
  которое реально уходит в AI-фильтр наравне с текстовым критерием) и
  OpenAI API-ключ (общая настройка уровня аккаунта, не привязана к
  конкретному фильтру).
- **History** — отдельная вкладка (не окно с Main): тумблер **"Use
  history"** (включён по умолчанию — при выключении находки за текущий
  прогон не пишутся в IndexedDB и уже виденные ссылки не отсеиваются
  между прогонами), счётчик "Total found", список всех найденных
  сущностей за всё время: заголовок, ссылка, описание (если источник
  двухфазный, обрезанное), доп. поля (компания/локация/теги — что собрал
  конкретный парсер), бейдж, по какому фильтру сущность сюда попала
  (**Keyword filter: ...** / **AI filter: ...** — если сработали оба,
  показываются оба бейджа), кнопка очистки истории.
- Кнопка **Logs** на Main (рядом со "Scan now"/"Stop") открывает окно
  технического журнала прогонов: что сканировалось, сколько ссылок
  найдено, сколько было новых, сколько прошло через keyword/minus-word/AI
  фильтры (и что именно было отсеяно, по заголовкам), ошибки отдельных
  сущностей. Полезно при отладке своего парсера и при настройке фильтров.
- Найденные элементы сохраняются в IndexedDB и переживают перезапуск браузера.
- Дедупликация — по нормализованному URL (без `utm_*` и трекинг-параметров).

## Набор собираемых данных

По умолчанию для каждой найденной сущности сохраняется:

- `title`, `url` — обязательные;
- `meta` — короткая строка для списка (например, компания · локация);
- `description` — полный текст (если источник двухфазный и отдаёт
  детальную страницу);
- `postedAt` — дата публикации на сайте-источнике (не путать с тем, когда
  расширение её нашло);
- `details` — произвольные доп. поля источника (`company`, `location`,
  `tags` и т.п. — какие уместны конкретному сайту).

Схема намеренно generic (не только про вакансии) — `details` это
свободный набор ключ-значение, который каждый парсер заполняет тем, что
имеет смысл для его темы.

## Как добавить свой источник

Два пути — выбирайте под ситуацию.

### Быстрый путь: попап "+ Add search" на вкладке Searches

Не требует правки файлов и перезагрузки расширения. Откройте попап, укажите:

- **Name** — короткий идентификатор источника;
- **List URLs** — одна или несколько страниц списка (каждая с новой
  строки — например разные регионы/фильтры одного сайта), все на одном
  домене.

Правила поиска ссылок и сбора данных захардкожены в
`parsers/searches.js`. Попап хранит только имя и URL сохранённого поиска;
вставлять JavaScript или включать "Allow User Scripts" не нужно.

### Постоянный путь: файл парсера в `parsers/`

Годится, если хотите, чтобы источник жил в репозитории (под git, для
переиспользования участниками воркшопа) — 3 шага:

1. Скопируйте `parsers/example-single-phase.js` (если все данные есть на
   странице списка) или `parsers/example-two-phase.js` (если нужно ещё
   открывать страницу каждой сущности за описанием) в новый файл, например
   `parsers/my-site.js`. Поменяйте `originPattern` и `listUrls` (массив —
   можно указать несколько URL списка, например разные регионы/фильтры
   одного сайта) на значения вашего сайта, перепишите `extractLinks` (и,
   если нужно, `extractDetail`) под реальную вёрстку — см. контракт в
   `parsers/parser-interface.js`.
2. Зарегистрируйте парсер в **двух** реестрах — оба нужны, это не
   альтернативы:
   - `parsers/index.js`:
     ```js
     import * as mySite from './my-site.js';
     // ...
     'my-site': {
       originPattern: mySite.originPattern,
       listUrls: mySite.listUrls,
       extractLinks: mySite.extractLinks,
       extractDetail: mySite.extractDetail, // опустите для однофазного парсера
     },
     ```
   - `parsers/searches.js` (`SUPPORTED_SOURCES`) — **это тот реестр, по
     которому попап "+ Add search" реально проверяет домен**; без записи
     здесь парсер не заработает через UI, даже если он есть в
     `parsers/index.js`:
     ```js
     import * as mySite from './my-site.js';
     // ...
     'my-site': {
       label: 'My Site',           // как источник подписан в UI
       origin: 'https://my-site.com', // без пути — сверяется URL.origin
       originPattern: mySite.originPattern,
       extractLinks: mySite.extractLinks,
       extractDetail: mySite.extractDetail,
     },
     ```
3. На вкладке "Searches" добавьте Search с List URL на новом домене (или
   нажмите **"Scan now"**, если URL уже сохранён где-то) — Chrome спросит
   разрешение на доступ к новому домену (один раз). Разрешите — и парсер
   заработает.

Правка `manifest.json` не требуется: доступ к новым доменам расширение
запрашивает динамически через `optional_host_permissions`.

**Важно:** `extractLinks`/`extractDetail` выполняются не в контексте
расширения, а прямо внутри вкладки с целевой страницей (через
`chrome.scripting.executeScript`) — это обычные функции без замыканий на
внешние переменные, видят только `document`/`window` этой страницы.
Прежде чем их писать — откройте сайт в браузере и проверьте селекторы в
консоли DevTools (`document.querySelectorAll(...)`), не угадывайте.

### Добавляете парсер с помощью Claude Code / Codex?

Если у вашего AI-инструмента есть доступ к браузеру (расширение
`claude-in-chrome`, встроенный browser-tool у Codex и т.п.) — попросите
его использовать именно этот доступ, чтобы сначала открыть целевой сайт
и посмотреть реальную структуру данных (Network-запросы или DOM), а не
угадывать поля по памяти. Подробный чек-лист и карта "что обычно нужно
поменять и где" — в [AGENTS.md](AGENTS.md).

## AI-фильтр (опционально)

AI-фильтр использует OpenAI Chat Completions API и требует API-ключ.
Расширение полностью работает и без него — на обычных фильтрах по словам.
Если у сущности есть `description`, его сниппет тоже передаётся модели —
не только заголовок. Если на вкладке "Setting" заполнен текст резюме —
он тоже передаётся модели вместе с текстовым критерием (вкладка
"Filters"), чтобы фильтр учитывал ваш профиль, а не только ключевые
слова, которые вы написали в критерии.

Получить ключ: https://platform.openai.com/api-keys

Ключ вводится на вкладке **"Setting"** (не "Filters" — это общая
настройка аккаунта, а не критерий конкретного фильтра) и хранится только
локально в браузере (`chrome.storage.local`), никуда кроме
`api.openai.com` не отправляется.

`.txt`-вложений текст читается автоматически при выборе файла.

## Working with an AI coding agent

Use these short prompts when you want an AI coding agent to work with this
repository.

### Project orientation

> Study this project and explain how to add a new website source. Read
> AGENTS.md and existing sources. Do not change code yet.

### Site analysis

> Analyze the site at the provided URL. Identify item cards, extractable fields,
> a stable unique identifier, and pagination. If browser access is available,
> inspect the live page. Do not write code yet.

### Add a source

> Add support for the site described in docs/new-source-request-template.md.
> First inspect the live site if browser access is available, then show the
> plan, implement it through the existing parser architecture, and run available
> checks.
